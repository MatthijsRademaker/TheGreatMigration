# planning-window Specification

## Purpose
Define a single frontend source of truth for the move-plan date range so all date-dependent views derive their rendered content from one canonical constant, eliminating hard-coded date literals scattered across feature views.

## ADDED Requirements

### Requirement: The planning window SHALL be defined as a single shared TypeScript module

The planning window SHALL be implemented in `frontend/src/shared/lib/planWindow.ts` as a plain TypeScript module exporting immutable constants and pure derivation functions. The module SHALL export `PLAN_WINDOW_START` and `PLAN_WINDOW_END` as ISO 8601 date string constants (defaulting to `"2026-07-05"` and `"2026-08-13"`). The module SHALL export `planWindowDays` as a derived array of inclusive day objects `{date: Date, label: string, dateString: string}`. The module SHALL export `planWindowDayCount` as the inclusive count of days between start and end. The module SHALL export `formatPlanDayLabel(date: Date): string` formatting a date as a compact human-readable label (e.g., `"Sun 5 Jul"`).

#### Scenario: Default date range matches the product example
- **WHEN** the module is imported with its default constants
- **THEN** `PLAN_WINDOW_START` equals `"2026-07-05"` and `PLAN_WINDOW_END` equals `"2026-08-13"`

#### Scenario: Derived day count reflects inclusive range
- **WHEN** `planWindowDayCount` is evaluated against the default constants
- **THEN** it returns `40` (27 days in July plus 13 days in August, both inclusive)

#### Scenario: Derived day list spans the full inclusive range
- **WHEN** `planWindowDays` is evaluated
- **THEN** its length equals `planWindowDayCount`, the first entry has `dateString` equal to `PLAN_WINDOW_START`, and the last entry has `dateString` equal to `PLAN_WINDOW_END`

#### Scenario: Date label formatting produces compact day-of-week date labels
- **WHEN** `formatPlanDayLabel` is called with a Date representing 2026-07-05
- **THEN** the returned string contains both the day number (`"5"`) and the abbreviated month (`"Jul"`)

### Requirement: CalendarView SHALL derive its day columns from the planning window

`frontend/src/calendar/CalendarView.vue` SHALL remove its hard-coded five-element `moveDays` array. The rendered day columns SHALL be generated by iterating over `planWindowDays`. Each column SHALL display the formatted date label from `planDay.label`. The existing per-column narrative `focus` and staffing `helpers` fields SHALL be replaced with a generic placeholder description. The grid layout SHALL use responsive column classes that adapt to the planning window size rather than a fixed `md:grid-cols-5`.

#### Scenario: Day columns match the planning window size
- **WHEN** `CalendarView` renders with the default 40-day planning window
- **THEN** the view renders exactly 40 day-column elements, each labeled with a date in the planning window range

#### Scenario: CalendarView grid adapts to arbitrary window sizes
- **WHEN** the planning window is configured to span a different number of days
- **THEN** the number of rendered columns changes accordingly without breaking the layout

### Requirement: HomeView Move days summary SHALL derive from the planning window

`frontend/src/home/HomeView.vue` SHALL replace the hard-coded `"5"` value for the "Move days" summary card with the value of `planWindowDayCount`. The card's label ("Move days"), description ("Scheduled working days in the plan"), and icon (`CalendarDaysIcon`) SHALL remain unchanged.

#### Scenario: Move days summary reflects the planning window
- **WHEN** `HomeView` renders with the default 40-day planning window
- **THEN** the "Move days" summary card displays `"40"` as its value

#### Scenario: Other HomeView summary cards are unchanged
- **WHEN** `HomeView` renders after the planning-window integration
- **THEN** the "Available today" card displays `"6"` and the "Under-staffed" card displays `"3"` as before

### Requirement: PeopleView SHALL defer planning-window integration

`frontend/src/people/PeopleView.vue` SHALL remain unchanged in this change. Its weekday availability strings represent availability patterns, not date-derived data. A code comment SHALL be added noting that future integration with `planWindowDays` is expected when real availability CRUD is built. The `planWindow` module header comment SHALL document its exports for future PeopleView consumers.

#### Scenario: PeopleView placeholder data is preserved
- **WHEN** `PeopleView` renders after the planning-window change
- **THEN** the helper availability cards display the same weekday strings (`"Mon, Wed, Fri"`, `"Tue, Wed"`, etc.) as before

### Requirement: The hello-world-integration spec SHALL be amended for the dynamic Move days card

The canonical spec at `openspec/specs/hello-world-integration/spec.md` SHALL be amended so that the "Other HomeView sections are preserved" scenario no longer requires the "Move days" summary card to render with static content. The "Available today" and "Under-staffed" cards, plus the "Today's plan" and "Move notes" sections, SHALL continue to render with their existing static content.

#### Scenario: Move days card is exempted from static-content requirement
- **WHEN** the amended hello-world-integration spec is validated
- **THEN** the "Move days" card is no longer required to display a static `"5"` value

#### Scenario: Remaining static cards are still protected
- **WHEN** the amended hello-world-integration spec is validated
- **THEN** the "Available today" and "Under-staffed" cards must still render with their existing static content (`"6"` and `"3"` respectively)

### Requirement: Planning window derivation logic SHALL be covered by tests

Unit tests in `frontend/tests/planWindow.test.ts` SHALL validate the constants, day count, day list, and label formatting. The existing route render test in `frontend/tests/app-routes-render.test.ts` SHALL be extended with an assertion that the `/calendar` route renders a number of day columns matching `planWindowDayCount`.

#### Scenario: Unit tests validate inclusive range derivation
- **WHEN** `planWindow.test.ts` unit tests run
- **THEN** all assertions pass for the default date range constants, derived day count, derived day list boundaries, and label formatting

#### Scenario: Cross-view render test proves propagation
- **WHEN** the route render test runs for `/calendar`
- **THEN** the rendered output contains exactly `planWindowDayCount` day-column elements

#### Scenario: Precommit checks pass
- **WHEN** `scripts/precommit-run` is executed
- **THEN** all lint, type-check, and test checks pass without errors
