## Why

The `PeopleAvailability` shared component currently renders a generic `CardHeader` with descriptive subtitle text and a summary row (`X of Y available today`) that do not appear in the canonical design references (`designs/home-page.png`, `designs/components.png`). The matrix anatomy is verbose — a "Person" column header, abbreviated day labels, full-text status badges, and an inline legend inside the scroll container — while the design references show a denser, title-only card with date+weekday column headers, compact avatar/name rows, soft semantic status pills, and a footer legend. The existing canonical specs (`dashboard-people-availability`, `people-availability-integration`) and the design-system contract (`docs/design-system-v2.md`) define the authoritative component contract, but the current change artifacts are placeholders containing no text-only implementation brief.

## What Changes

- **Shared component (`frontend/src/people/PeopleAvailability.vue`)**: Strip the `CardDescription` and summary row from the template. Compact the matrix by removing the "Person" column header, switching day-column headers to date+weekday format (e.g., "Sun 5 Jul"), and tightening status badge sizing. Move the legend from inside the `overflow-x-auto` container to a single footer row below the matrix. Preserve all editable-mode behaviour (Popover triggers, Actions column, Delete buttons, confirmation dialogs).
- **Shared adapter (`frontend/src/shared/composables/usePeopleAvailability.ts`)**: Stop producing the hard-coded `description` prop in `adaptToComponentProps`. Remove `availableToday` and `totalPeople` from the adapted `PeopleAvailabilityProps` output (the raw dashboard response remains available via `rawData` for future use).
- **Consumer views**: `HomeView.vue` and `PeopleView.vue` continue to spread adapter data via `v-bind="availabilityData"` — no template changes required. The `/people` route chrome (create form, pagination bar, loading/error/empty cards) is explicitly unchanged.
- **Frontend tests**: Update assertions in `PeopleAvailability.test.ts` (remove summary row and description assertions; update date header format and column header expectations; keep all editable-mode tests intact), `PeopleView.test.ts` (verify editable mode preserved without description or summary), and `app-routes-render.test.ts` (replace "6 of 8 available today" assertion for `/people` with legend footer and matrix structure assertions).
- **Spec artifact**: A new text-only design-alignment spec is created at `openspec/changes/task-15d3e474-3338-45a3-a7ff-51246d719e92/specs/people-availability-design-alignment/spec.md` following the established pattern from `task-management-design-alignment` and `daily-schedule-design-alignment`.

## Impact

- **Affected specs**: New `people-availability-design-alignment` spec added under the change directory; no existing canonical specs (`dashboard-people-availability`, `people-availability-integration`) are modified.
- **Affected code**: `frontend/src/people/PeopleAvailability.vue` (template only), `frontend/src/people/types.ts` (remove `description`, `availableToday`, `totalPeople` from `PeopleAvailabilityProps`), `frontend/src/shared/composables/usePeopleAvailability.ts` (adapter output), `frontend/tests/people/PeopleAvailability.test.ts`, `frontend/tests/people/PeopleView.test.ts`, `frontend/tests/app-routes-render.test.ts`.
- **Affected routes**: `/` (home dashboard, read-only) and `/people` (management, editable) — both render the shared component and pick up its visual changes automatically.
- **No impact on**: Backend endpoints, OpenAPI contract, persistence, canonical availability status vocabulary (`available`, `busy`, `partial`, `off`), `/people` route chrome (create form, pagination bar, loading/error/empty state cards), or the Avatar primitive.
